# 后训练项目服务器运行——异常问题汇总

> 项目：QLoRA 中文指令微调（Qwen2.5-7B-Instruct）
> 服务器：4×RTX 4090, Ubuntu, CUDA 驱动 12.2, conda
> 时间跨度：2026-06-10 16:59 ~ 2026-06-11 19:13
> 记录目的：避免同类问题再次踩坑

---

## 一、环境搭建阶段（06-10 17:00 ~ 20:49）

### 问题 1：pip 安装 PyTorch 卡住

- **现象**：`pip install torch` 长时间无响应，卡在下载阶段
- **根因**：默认 PyPI 源在国内下载速度极慢
- **解决**：切换清华源 `pip install torch -i https://pypi.tuna.tsinghua.edu.cn/simple`
- **教训**：**国内服务器装包，第一步就换国内镜像源**

### 问题 2：PyTorch CUDA 版本与驱动不兼容

- **现象**：先装了 CUDA 12.4 版 PyTorch，但 `torch.cuda.is_available()` 返回 False
- **根因**：服务器 NVIDIA 驱动只支持到 CUDA 12.2，PyTorch 2.6+cu124 要求更高版本驱动
- **解决**：降级安装 `PyTorch 2.5.1+cu121`（pip install torch==2.5.1 --index-url https://download.pytorch.org/whl/cu121）
- **教训**：**装 PyTorch 前先 `nvidia-smi` 查看驱动支持的最高 CUDA 版本，PyTorch 的 CUDA 版本必须 ≤ 驱动版本**

### 问题 3：transformers 5.10.2 与 PyTorch 2.5.1 不兼容

- **现象**：`ImportError` / `AttributeError`，报错需要 `torch.float8_e8m0fnu` 类型（这是 PyTorch 2.6+ 才有的 dtype）
- **根因**：transformers 5.10.2 使用了 PyTorch 2.6 才引入的 float8 数据类型
- **解决**：降级 `transformers==4.46.3`
- **教训**：**PyTorch 版本锁死后，必须检查 transformers 对 PyTorch 的最低版本要求，不能装最新的 transformers**

### 问题 4：trl 与 transformers 版本互斥

- **现象**：`trl 1.5.1` 要求 `transformers>=4.56.2`，而上面已将 transformers 降级到 4.46.3
- **根因**：trl 高版本依赖高版本 transformers，形成版本锁链
- **解决**：最终版本组合 `transformers==4.46.3 + trl==0.12.2 + peft==0.13.2`
- **教训**：**Python ML 生态的版本兼容是典型的钻石依赖问题，降级一个包后必须检查整个依赖链。推荐用 `pipdeptree` 或 `pip check` 验证**

### 问题 5：HuggingFace 模型下载极慢

- **现象**：服务器上下载 Qwen2.5-7B（~15 GB），速度仅 0.72 MB/s，预计 14 天
- **尝试方案**：
  - `hf-transfer`：速度提升至 1-3 MB/s 但不稳定
  - `modelscope`：速度反而降至 286-324 kB/s
  - `aria2` 多线程：因 HuggingFace XET 存储的 `Content-Length` 与实际文件大小不一致，大量 `Invalid range header` 错误，3/4 文件下载失败
- **解决**：改用本地 Windows 下载 + scp 上传
- **教训**：**服务器网络不好的情况下，不要死磕在线下载。本地下 + scp 上传往往更可靠。aria2 对 HuggingFace XET 存储的分片下载不兼容，不要用**

### 问题 6：PowerShell Invoke-WebRequest 遇 308 重定向失败

- **现象**：用 PowerShell `Invoke-WebRequest` 下载 HuggingFace 小文件时报 308 重定向错误
- **根因**：PowerShell 的 `Invoke-WebRequest` 默认不跟随 308 重定向
- **解决**：改用 `curl` 下载（curl 自动跟随重定向）
- **教训**：**下载 HuggingFace 文件优先用 curl，不要用 PowerShell 原生命令**

### 问题 7：curl 下载超时（curl error 28）

- **现象**：4 个大文件并行下载时，文件 2 首次连接 hf-mirror.com 超时（21042ms），下载为 0 字节
- **根因**：校园网到 hf-mirror 的连接不稳定，部分请求超时
- **解决**：重新启动该文件的下载命令，重试成功
- **教训**：**大文件下载必须加超时重试机制，建议 `curl --retry 3 --retry-delay 5`**

## 二、项目运行阶段（06-10 21:01 ~ 06-11 19:13）

### 问题 8：scp 上传的模型文件未被代码识别

- **现象**：用 WinSCP 将模型文件上传到 `models/Qwen2.5-7B/` 目录后，运行 smoke_test.py 仍从 HuggingFace 自动下载模型（速度 401 kB/s）
- **根因**：代码中模型路径通过 `from_pretrained("Qwen/Qwen2.5-7B-Instruct")` 加载，HuggingFace 默认从远程仓库下载到 `~/.cache/huggingface/`，而不是读本地 `models/` 目录
- **解决**：需要修改代码中的模型路径指向本地目录，或设置 `cache_dir` 参数指向本地模型目录
- **教训**：**HuggingFace 的 `from_pretrained()` 优先从缓存/远程加载，不是自动读本地目录。必须明确传入本地路径，如 `from_pretrained("/path/to/local/models/Qwen2.5-7B")`**

### 问题 9：config.yaml active_profile 被意外改回 local

- **现象**：E2 的 C-Eval 评测结果显示 `base_model: "Qwen/Qwen2.5-1.5B-Instruct"`（1.5B 小模型），而非预期的 7B 模型
- **根因**：`run_ablation.py` 的子进程可能未继承正确的 config 状态，或 config.yaml 的 `active_profile` 被某次操作改回了 `local`
- **影响**：E2 实验实际在 1.5B 模型上运行，实验结果无效
- **教训**：**运行批量实验前，必须验证 config 的 active_profile 是正确的。建议在脚本启动时打印当前 profile 和模型名称作为检查点**

### 问题 10：NCCL P2P 通信错误（多卡训练）

- **现象**：多卡训练时出现 NCCL 相关的 P2P 通信错误
- **根因**：部分 GPU 之间的 P2P 直连不可用（PCIe 拓扑/NVLink 配置问题）
- **解决**：设置环境变量 `NCCL_P2P_DISABLE=1` 禁用 P2P，改用共享内存通信（性能略降但稳定）
- **教训**：**多卡训练遇到 NCCL 错误时，先试 `NCCL_P2P_DISABLE=1`。如果仍有问题再查 `NCCL_DEBUG=INFO` 看详细日志**

### 问题 11：SFTConfig 参数名与 trl 版本不匹配

- **现象**：训练脚本使用 `SFTConfig` 的某些参数时报 `TypeError`（unexpected keyword argument）
- **根因**：不同版本 trl 的 `SFTConfig` 参数名不同（如 `max_seq_length` vs `max_length`，`packing` 参数的默认值变化等）
- **解决**：对照 trl 0.12.2 的文档调整参数名
- **教训**：**trl 的 API 在不同小版本间变化很大，升级/降级 trl 后必须逐个检查训练参数是否兼容**

### 问题 12：多卡训练 device_map 不一致

- **现象**：LoRA 合并/推理时模型加载到了错误的 GPU 设备
- **根因**：训练时用 `device_map="auto"` 自动分配多卡，但推理时没有用相同的 device_map，导致张量设备不一致
- **解决**：推理时也使用 `device_map="auto"` 或明确指定设备
- **教训**：**训练和推理的 device_map 必须一致。`device_map="auto"` 虽然方便但不一定最优，生产环境建议手动指定**

### 问题 13：run_ablation.py 数据准备只生成 max_size 的数据

- **现象**：E2（5000 条）训练失败，报 `FileNotFoundError: train_5000.jsonl` 不存在
- **根因**：`run_ablation.py` 的 prepare 步骤只生成了 `max_size=10000` 的数据文件（`train_10000.jsonl`），但实验计划中 E2 需要 `train_5000.jsonl`，脚本没有为每个 train_size 单独生成对应的数据文件
- **解决**：手动运行 `prepare_data.py` 补生成 5000 条数据
- **教训**：**消融实验脚本必须为每个 train_size 单独准备数据文件，不能假设大尺寸的数据文件能覆盖小尺寸。设计数据准备流程时要检查所有实验配置的数据需求**

### 问题 14：E3 holdout 评测 nan 崩溃（CUDA assert）

- **现象**：
  ```
  ../aten/src/ATen/native/cuda/TensorCompare.cu:110: _assert_async_cuda_kernel:
  Assertion `probability tensor contains either `inf`, `nan` or element < 0` failed.
  RuntimeError: CUDA error: device-side assert triggered
  ```
  holdout-base 评测 100% 通过，但 holdout-lora 评测在第一条就崩溃
- **根因**：E3 用 10000 条数据训练了 2 个 epoch，模型过拟合导致生成时 softmax 概率出现 nan/inf
- **可能修复**：
  1. 降低学习率或减少 epoch 数
  2. 加入 `temperature` 参数避免极端概率
  3. 生成时设置 `do_sample=False`（贪心解码，不经过 softmax 采样）
  4. 调试时先 `CUDA_LAUNCH_BLOCKING=1` 定位具体触发位置
- **教训**：**大模型 + 小数据 + 多 epoch = 过拟合 → 生成 nan。消融实验中数据量最大的实验尤其要关注 epoch 数和 learning rate 的搭配。建议在训练过程中监控 loss 是否异常下降到接近 0**

## 三、根因归类与防坑清单

| 根因类型 | 涉及问题 | 防坑策略 |
|---------|---------|---------|
| **版本兼容** | #2, #3, #4, #11 | 先锁定 PyTorch → 再选 transformers → 最后选 trl/peft；装完跑 `pip check` |
| **网络/下载** | #1, #5, #6, #7 | 国内换源；大模型本地下+scp上传；用 curl 不用 PowerShell；加 retry |
| **路径/配置** | #8, #9, #12 | `from_pretrained` 传绝对路径；运行前打印 config profile；训练推理 device_map 一致 |
| **代码设计缺陷** | #13 | 数据准备脚本为每个 size 单独生成文件 |
| **训练超参** | #14 | 大数据量+多 epoch 注意过拟合；监控 loss；生成时用贪心解码兜底 |
| **多卡通信** | #10 | NCCL 报错先试 `NCCL_P2P_DISABLE=1` |

## 四、推荐的标准启动检查清单

1. `nvidia-smi` 确认驱动 CUDA 版本
2. `python -c "import torch; print(torch.cuda.is_available(), torch.version.cuda)"` 确认 PyTorch CUDA 可用
3. `pip check` 验证依赖无冲突
4. 确认 `config.yaml` 的 `active_profile` 和模型名称
5. 确认模型文件在本地路径存在且完整
6. 多卡训练设置 `NCCL_P2P_DISABLE=1`
7. 训练脚本中加入 config profile 打印，确认模型正确
8. 监控训练 loss 曲线，异常下降立即预警
